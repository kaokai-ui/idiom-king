# 成語接龍 Web 遊戲開發規劃

## 1. 專案目標

要做一款以成語接龍為核心玩法的 Web 遊戲，畫面風格可參考你提供的手機直式介面：

- 每一關從成語題庫中挑出 5 到 10 個成語。
- 成語會以「上、下、左、右」的方式交錯排列在棋盤上。
- 棋盤上先顯示空格，玩家要從下方的字牌區點字填入。
- 玩家操作流程：
  1. 先點選上方某個空格。
  2. 再點選下方某個中文字。
  3. 該字會填入空格。
- 當所有空格都填滿，且所有成語的排列與答案完全正確，就過關並進入下一關。

這份文件的目的是先把「怎麼做」拆清楚，方便後續正式開發。

## 2. 建議開發策略

建議先做 `MVP`，再做正式版。

原因是這類遊戲真正困難的地方，不是畫面，而是：

- 題庫整理
- 關卡生成
- 格子交叉合法性檢查
- 玩家填字流程與過關判定

所以最穩的做法是：

1. 先完成一版可玩的固定關卡系統。
2. 再把固定關卡升級成可自動生成關卡。
3. 最後再補美術、音效、提示、道具、分數等系統。

## 3. 現有可利用素材

idiom.txt

## 4. 建議技術選型

如果這款遊戲要做成一般網頁版本，建議使用：

- 前端框架：`React + TypeScript + Vite`
- 樣式：`CSS / SCSS` 或 `Tailwind CSS`
- 狀態管理：先用 React state 即可，後期再看要不要加 `Zustand`
- 部署：`Vercel`、`Netlify` 或自架靜態網站

這樣的原因是：

- 棋盤格子、字牌區、關卡切換都很適合用元件化方式處理。
- TypeScript 比較適合管理關卡資料結構。
- Vite 啟動快，適合反覆測試遊戲互動。

如果你只想先做超快速原型，也可以先用：

- `HTML + CSS + Vanilla JavaScript`

但若之後要補關卡、動畫、提示、手機版適配，React 會比較好維護。

## 5. 核心系統拆解

這個遊戲可以拆成 6 個主要模組：

1. 題庫資料模組
2. 關卡生成模組
3. 棋盤渲染模組
4. 玩家填字互動模組
5. 驗證與過關模組
6. 關卡進度與難度模組

下面逐一說明。

## 6. 開發步驟

### 階段 1：先定義遊戲規則

第一步先把規則寫死，避免做到一半一直改資料結構。

建議先定義：

- 成語來源為 4 個字~9個字。
- 一關目標數量是否固定為 5、6、7...10，還是隨難度變化。
- 格子是否允許空白障礙格。
- 玩家填錯時要不要立即提示。
- 下方字牌是否可重複使用。
- 是否支援覆蓋已填字。
- 是否提供「清空」、「撤回」、「提示」功能。

建議 `MVP` 規則：

- - 一關先做 5 個成語。
- 允許成語彼此交叉共用同一個字。
- 玩家可以覆蓋重填。
- 所有格子填完後才進行正確性檢查。
- 先不做計時、不做金幣、不做道具。

### 階段 2：整理成語題庫

成語題庫是idiom.txt可以轉成乾淨、可用的 JSON。

### 2.1 要整理的欄位

每個成語至少要保留：

- `id`
- `text`：成語全文，例如 `一心一意`
- `chars`：拆成字陣列，例如 `["一", "心", "一", "意"]`
- `definition`：成語解釋
- `pinyin` 或 `bopomofo`：可選
- `difficulty`：可後補
- `source`：資料來源，可選

建議輸出格式：

```json
[
  {
    "id": "idiom_0001",
    "text": "一心一意",
    "chars": ["一", "心", "一", "意"],
    "definition": "心思專一，毫無他念。"
  }
]
```

### 2.2 題庫清洗規則

要做以下過濾：

- 移除空值資料
- 移除不是 4 字的詞條
- 移除重複成語
- 移除生僻到不適合一般玩家的條目
- 如果同字形但解釋不同，先只保留一筆

### 2.3 補充前端會需要的欄位

可以在清理資料時順便產生：

- `uniqueChars`：該成語的去重字集
- `charCountMap`：每個字出現幾次
- `hint`：顯示給玩家的簡短提示

例如：

```json
{
  "text": "自得其樂",
  "chars": ["自", "得", "其", "樂"],
  "charCountMap": {
    "自": 1,
    "得": 1,
    "其": 1,
    "樂": 1
  }
}
```

### 階段 3：先做固定關卡，不急著做自動生成

這一步很重要。

雖然目標是「從題庫抓 5 到 10 個成語」，但實務上建議：

1. 先手工做 10 到 20 關固定關卡。
2. 把遊戲流程、UI、判定邏輯做穩。
3. 再投入自動生關卡演算法。

原因：

- 自動生關卡是最耗時間的部分。
- 如果先做演算法，畫面與操作還沒定，會很難 debug。
- 固定關卡可以先驗證玩家體驗。

### 3.1 固定關卡資料格式

每一關建議存成：

```json
{
  "id": 1,
  "rows": 8,
  "cols": 8,
  "idioms": [
    {
      "text": "一心一意",
      "direction": "horizontal",
      "startRow": 2,
      "startCol": 1
    },
    {
      "text": "意氣風發",
      "direction": "vertical",
      "startRow": 2,
      "startCol": 4
    }
  ],
  "charBank": ["一", "心", "一", "意", "氣", "風", "發"]
}
```

### 3.2 關卡產出結果

固定關卡資料要能推導出：

- 哪些格子可填
- 每個格子的正確答案
- 哪些格子是交叉共用
- 下方字牌池有哪些字

### 階段 4：設計自動生成關卡演算法

當固定關卡流程跑順後，再開始做真正的自動生成。

### 4.1 自動生成的基本邏輯

可用下面流程：

1. 從題庫隨機選一個成語當起點。
2. 把它放進棋盤中央。
3. 從剩餘成語中找可與現有字交叉的成語。
4. 嘗試用上、下、左、右的方式插入棋盤。
5. 每放入一個成語，就檢查是否合法。
6. 成功放滿 5 到 10 個成語後，輸出本關資料。
7. 如果嘗試次數太多仍失敗，就重開新局或改選別組成語。

### 4.2 合法性檢查規則

每次放成語時都要檢查：

- 是否超出棋盤邊界
- 新字是否與既有字衝突
- 交叉位置是否為同一個字
- 是否會形成不該存在的額外詞
- 是否和其他成語黏太近
- 整關是否保持連通，不要出現分離區塊

### 4.3 演算法建議

初版建議先用：

- `回溯法（Backtracking）`

流程是：

- 先放一個成語
- 再試第二個
- 若第三個放不下，就回退重試

這種做法雖然不一定最快，但最容易做對，也最適合你現在這個遊戲規模。

### 4.4 難度控制方式

可以透過以下條件控制難度：

- 成語數量由 5 提升到 10
- 棋盤變大
- 交叉點變多
- 下方干擾字變多
- 提示變少
- 生僻成語比例提高

### 階段 5：前端畫面與互動設計

畫面可以拆成這幾區：

1. 頂部列
2. 棋盤區
3. 提示區
4. 字牌區
5. 功能按鈕區

### 5.1 頂部列

顯示：

- 返回按鈕
- 音效按鈕
- 關卡編號
- 金幣或分數

### 5.2 棋盤區

棋盤每格會有幾種狀態：

- `disabled`：不可用空格
- `empty`：可填但尚未填字
- `filled`：已填字
- `selected`：目前玩家選中的格子
- `correct`：過關後或檢查後正確
- `wrong`：檢查後錯誤

### 5.3 提示區

建議先顯示：

- 本關提示文字
- 目前剩餘未填格數

後續可再加：

- 當前成語釋義
- 錯誤提示
- 使用提示道具按鈕

### 5.4 字牌區

下方字牌區的需求：

- 每個字都能點擊
- 已使用字可標記為已放置
- 若同字出現兩次，要能區分成兩張牌
- 若玩家取消填寫，字牌要能回收

### 階段 6：實作玩家填字流程

這一段是遊戲手感的核心。

建議流程：

1. 玩家點棋盤格子。
2. 系統記錄 `selectedCell`。
3. 玩家點下方字牌。
4. 系統把字填入 `selectedCell`。
5. 該字牌標記為已使用。
6. 自動移動到下一個空格，或等待玩家重新點格。

### 6.1 需要管理的狀態

前端至少要有以下狀態：

- `currentLevel`
- `board`
- `selectedCell`
- `charBank`
- `placedChars`
- `isLevelComplete`

### 6.2 建議資料結構

```ts
type Cell = {
  row: number;
  col: number;
  isActive: boolean;
  answer: string;
  currentValue: string;
};
```

```ts
type CharTile = {
  id: string;
  value: string;
  used: boolean;
};
```

### 階段 7：完成判定與過關流程

過關判定建議在「所有可填格都已填字」後觸發。

判定流程：

1. 掃描所有有效格子。
2. 比對 `currentValue` 是否等於 `answer`。
3. 若全部正確，顯示過關動畫。
4. 解鎖下一關。
5. 載入下一關資料。

### 7.1 錯誤處理方式

有兩種常見做法：

- 寬鬆型：填完才一次檢查
- 即時型：每填一字就立刻判定

建議 `MVP` 先做寬鬆型，因為：

- 邏輯較單純
- 比較接近填字遊戲體驗
- 不會讓玩家每一步都被打斷

### 階段 8：關卡進度與遊戲內容

當基本玩法完成後，再加成長內容。

可以加入：

- 關卡地圖
- 解鎖機制
- 星等評分
- 提示道具
- 洗牌字牌
- 看廣告拿提示
- 每日挑戰

如果想先快速上線，這些都可以延後。

## 7. 建議的實作順序

下面是一個最實際的落地順序。

### 第 1 步：建立前端專案骨架

要做的事：

- 建立 Vite 專案
- 建好手機直式版面
- 先切出靜態畫面

完成標準：

- 畫面有上方資訊列
- 中間有棋盤
- 下方有字牌按鈕

### 第 2 步：建立固定關卡資料

要做的事：

- 手工定義 3 到 5 關
- 先不串 Excel
- 直接用 JSON 寫死

完成標準：

- 可以讀取關卡
- 棋盤能正確顯示空格位置

### 第 3 步：完成填字互動

要做的事：

- 點空格選取
- 點字牌填字
- 支援修改與清除

完成標準：

- 玩家能完整操作一關

### 第 4 步：完成過關判定

要做的事：

- 所有格子填滿後自動檢查
- 正確就顯示過關
- 能切到下一關

完成標準：

- 至少能順玩 3 關

### 第 5 步：把題庫轉成 JSON

要做的事：

- 把 Excel 轉成乾淨的成語資料
- 寫過濾與清洗腳本

完成標準：

- 前端能載入成語題庫 JSON

### 第 6 步：實作自動生關卡

要做的事：

- 撰寫交叉配對邏輯
- 實作回溯放置
- 加入失敗重試機制

完成標準：

- 系統能自動生成合法關卡

### 第 7 步：補 UI、美術與音效

要做的事：

- 調整色彩、格子樣式、選取狀態
- 加入過關動畫
- 加入點擊音效

完成標準：

- 有接近正式遊戲的體驗

## 9. 測試重點

這款遊戲最需要測的不是 API，而是規則正不正確。

重點測試項目：

- 點空格後是否能正確選中
- 點字牌後是否填到正確位置
- 重複字牌是否會用錯
- 交叉成語是否共用同一格
- 填滿後是否正確判定成功或失敗
- 下一關載入時狀態是否有清乾淨
- 手機版版面是否不會跑掉

另外建議把以下邏輯寫成可單獨測試的函式：

- 產生棋盤
- 成語放置檢查
- 關卡完成檢查
- 字牌使用與回收邏輯

## 10. 風險與注意事項

這個遊戲最容易卡住的點有 4 個：

1. 自動生成出來的關卡不好玩
2. 成語雖然能交叉，但視覺上太亂
3. 字牌重複字很多時，玩家操作容易混亂
4. 題庫太大時，生成速度會變慢

所以建議優先採取下面策略：

- 第一版先做固定關卡
- 關卡生成器先離線產關，不一定要即時生成
- 難度先靠人工挑關，不要一開始就全自動
- 先確保手機直式版手感，再處理桌機版

## 11. 建議的第一版開發里程碑

### 里程碑 A：可操作原型

目標：

- 有 1 關
- 能點格填字
- 能判斷成功

### 里程碑 B：可玩的 MVP

目標：

- 有 10 關固定關卡
- 有提示文字
- 有過關流程

### 里程碑 C：題庫版

目標：

- 能從整理後的成語 JSON 讀資料
- 可批次建立關卡

### 里程碑 D：正式版

目標：

- 自動生成關卡
- 手機版 UI 完成
- 加入音效、動畫、提示、進度系統

## 12. 最後建議

如果要讓這個專案順利落地，最推薦的順序是：

1. 先做固定關卡版，確認玩法成立。
2. 再整理成語題庫，建立標準 JSON。
3. 再做自動生成器，而不是一開始就硬上演算法。
4. 最後才補正式遊戲常見的提示、金幣、音效、美術與進度系統。

這樣做的好處是：

- 可以最快看到可玩的版本
- 比較容易找出玩法問題
- 後續擴充時不容易整個重寫
